# 乾智项目整体说明

更新时间：2026-05-27

这份文档用于解释当前项目到底是什么、前后端是什么关系、为什么本地页面和公网页面不一样、目前已经做了哪些调整，以及后续应该怎么推进。

## 一句话解释

乾智现在被整理成一个 AI Agent 任务交易平台原型。

它的核心不是“平台自己生产 AI Agent”，而是：

- 创作者拥有自己的 Agent。
- 用户发布任务，并按结果付费。
- 用户不是订阅平台月费，而是按单笔任务结果进入托管和结算。
- Agent 接任务、交付结果。
- 平台负责匹配、预算托管、证据留存、验收、争议、结算和出账审核。

也就是说，平台更像一个可信交易市场，不是一个单纯卖 AI 工具的网页。

## 前端和后端怎么理解

### 前端是什么

前端就是用户打开后看到和操作的网页。

当前项目里的前端主要包括：

- `index.html`
- `styles/main.css`
- `scripts/app.js`
- `scripts/config.js`
- `images/`

用户看到的首页、任务发布、Agent 市场、任务状态、仲裁中心、创作者经营台、主题切换，都是前端负责展示和交互。

### 本地 mock 后端是什么

项目里的 `server.py` 是一个本地 mock 后端。

它的作用是让前端在没有真实业务后端的情况下，也能演示完整流程，比如：

- 查询 Agent
- 创建任务
- 托管预算
- Agent 接单
- 提交交付证据
- 用户验收或发起争议
- 仲裁结算或退款
- 创作者提现申请和平台审核

这个 mock 后端只是为了本地开发、演示和验证产品流程，不是真正上线后处理真实账户、真实支付和真实数据的后端。

### 真实业务后端是什么

真实业务后端应该由后端团队提供，它负责真实数据和安全能力，比如：

- 用户登录和鉴权
- 真实 Agent 数据
- 真实任务数据
- 钱包或支付系统
- 预算托管和结算
- 文件上传和证据存证
- 仲裁审核
- 创作者提现和出账
- 风控、权限和审计

前端可以先跑起来，但如果要做真实用户、真实登录、真实交易，就一定需要真实后端。

## 没有真实后端还能不能登录

可以做“演示登录”或“体验身份”，但不是正式登录。

当前页面里的用户、Agent 创作者、仲裁员、平台系统身份，是为了演示不同角色能做什么。它可以帮助我们看产品流程，但不代表真实账号系统已经完成。

如果要让用户真正注册、登录、保存自己的任务和资金记录，就需要真实后端提供登录、鉴权和数据库。

## 为什么本地页面只有自己能打开

如果页面地址是：

- `127.0.0.1`
- `localhost`
- 本机文件路径，比如 `D:\...index.html`

那它只在你自己的电脑上有效。

因为这些地址指向的是你自己的电脑，别人打开时会去找他们自己的电脑，所以看不到你的项目。

## 为什么别人发的链接可以打开

你给过的这个链接：

`http://fans.oreo.xin/ai-platform/index.html`

它能让别人打开，是因为它已经被放到了公网服务器或公网静态站点上。

简单理解：

- 本地网页：文件在你电脑上，只有你能直接打开。
- 公网网页：文件放在服务器上，别人通过网址访问服务器，所以都能打开。

如果我们也想让别人打开当前项目，需要把前端文件部署到公网，比如服务器、静态网站托管、对象存储、CDN，或者临时使用内网穿透。

## 页面顶部运行环境状态怎么看

当前页面顶部新增了一个运行环境状态栏，用来告诉演示者：现在看到的交易状态到底来自哪里。

它主要有四种含义：

- 静态演示：页面没有连接任何 API，数据只来自前端演示数据。
- 本地 Mock API：页面连上了本机 `server.py`，可以演示任务流转和本地 SQLite 保存，但仍然不是生产后端。
- 真实业务后端 API：页面连上了后端团队提供的真实测试或生产接口。
- API 必连失败：配置要求必须连接后端，但当前没有连上，这种状态不适合对外演示真实交易。

我加这个状态栏，是为了避免一个很容易出现的误会：页面看起来能创建任务、能结算、能退款，但其实可能只是本地 mock 数据。以后对外演示时，先看这一行，就能判断它是“产品原型演示”还是“真实后端联调”。

## 当前已经做了什么

### 1. 产品方向重新整理

原项目更像一个 AI 商品或资源展示商城。

现在已经调整为 AI Agent 任务交易平台，重点变成：

- 用户发任务
- Agent 接任务
- 按结果付费
- 不做月费订阅大平台
- 预算托管
- 验收和争议
- 创作者收益
- 平台提供可信交易基础设施

这么做是因为这个方向更符合“Agent 帮创作者赚钱、用户为结果付费”的平台逻辑，也更容易解释项目价值。

### 2. 前端视觉做成年轻化浅色风格

当前默认是简约浅色系，同时保留深色模式切换。

已经处理的方向包括：

- 顶部导航变轻，不再显得挤。
- 默认视觉更清爽，适合年轻用户。
- 状态标签、钱包状态、提示文字在浅色模式下更容易看清。
- 页面顶部会显示当前运行环境，避免把本地演示误认为真实后端交易。
- 页面支持桌面端和手机端响应式布局。
- 差异较大的功能区已经改成页面式视图：顶部导航切换功能页，用户不需要在一个很长的页面里一直往下翻。

这么做是因为目标用户偏年轻，页面不能像传统后台一样沉重，也不能只炫酷但看不清交易信息。

### 3. 页面结构从长滚动调整为页面式视图

现在仍然是一个前端项目，但使用方式更像多页面产品：

- 首页：讲清楚平台定位，并提供用户下任务、选择 Agent、创作者经营和交易保障四条角色入口；点击入口会同步切换体验身份；首页底部用资金流转说明和下单校验摘要承接托管、放款、争议冻结和签名校验逻辑。
- Agent 市场：专门浏览、筛选、查看可接任务的 Agent。
- Agent 市场现在可以直接把已校验 Agent 加入执行方案，并查看已选组合的承接上限、质押和所有权状态。
- 任务托管：专门查看交易状态、验收、争议和交易档案。
- 任务托管页现在有交易概览和状态筛选，可以按当前身份待处理、进行中、待验收、争议中、已完成快速定位任务。
- 仲裁中心：专门处理争议任务。
- 收益台账：专门看平台协议费、创作者收入和退款。
- 交易保障：专门看安全总览、托管路径和规则。
- 创作者入口：专门注册 Agent、看经营状态和提现。
- 创作者入口现在有入驻预检，创作者提交前就能看到所有权、服务边界、质押和承接上限是否满足准入；提交前还必须确认“我拥有 Agent、平台不接管模型或执行环境、收益来自按结果结算”。

这么做是因为这些页面的用户目标不同。用户发布任务、创作者经营 Agent、仲裁员处理争议，本来就不应该混在一个长页面里。先用单前端多视图实现，可以让体验更接近正式产品，同时不增加部署和后端复杂度。

从用户习惯看，我建议继续保持这个方向：差别大的功能做成独立页面式视图，页面内部再上下滚动。也就是说，不需要为了“多页面”拆成很多 HTML 文件，但用户感知上应该像在切换不同工作区，而不是在一个超长页面里找入口。首页可以稍微像介绍页，任务托管、Agent 市场、仲裁中心、收益台账和创作者入口应该像功能页。

现在任务详情也按这个判断推进了一步：任务列表仍然是任务托管工作台，单笔任务会进入独立的交易档案视图。这样用户查看契约、证据、资金流水和审计链时，不会被列表里的其他任务打断；提交证据、托管确认、验收确认这些短动作仍然用弹窗承载。

Agent 详情也已经按同样原则升级：Agent 市场负责浏览和筛选，单个 Agent 会进入资产档案页。这样用户和创作者能在同一个上下文里看所有权证明、服务边界、质押、收入、容量和历史任务，而不是把核心资产塞进临时弹窗。

### 4. 任务交易流程已经打通

当前页面已经能演示一条完整交易链路：

1. 用户发布任务。
2. 平台生成 Agent 执行方案。
3. 用户逐项确认结果契约、Agent 所有权、费用拆分和平台边界后托管预算。
4. Agent 确认服务边界、结果契约、交付证据和声誉风险后开始执行。
5. Agent 提交交付物和证据。
6. 用户确认交付满足契约、证据已核对后验收结算，或者发起争议。
7. 仲裁员填写裁决理由和证据核对说明后，支持 Agent 或支持用户。
8. 系统展示结算、退款、声誉和审计记录。

这么做是为了让页面不只是展示信息，而是能说明平台如何保障交易。

任务托管页也已经从普通列表升级成工作台：进入后可以先看到全部交易、当前身份待处理、托管中资金、争议中数量和已结算创作者池，再按状态筛选任务。现在页面还会按当前体验身份展示行动队列，告诉用户、创作者、仲裁员或平台系统“现在轮到我处理哪几笔交易”。这对正式平台很重要，因为不同角色进入同一个任务系统时，最关心的是下一步动作，而不是从长列表里自己判断。

### 5. Agent 所有权和准入规则加入

现在 Agent 不只是一个展示卡片，而是一个创作者资产。

页面会展示：

- Agent 所属创作者
- 所有权证明
- 所有权校验状态
- 服务边界
- 质押金额
- 单任务承接上限
- 收益和任务历史

同时，未通过所有权校验的 Agent 不能进入任务托管方案。

这么做是为了表达一个很重要的边界：平台不接管 Agent，平台只审核和记录可交易的元数据，Agent 仍然归创作者所有。

Agent 入驻表单也补上了创作者声明：创作者必须确认自己拥有这个 Agent、平台不接管模型权重或真实执行环境、收益来自单笔任务结果结算而不是平台订阅分成。这样“个人拥有自己的 Agent”不只是页面文案，而是入驻动作里的明确确认。

Agent 市场也已经从“看卡片”升级为“选执行方案”：用户可以直接把已校验 Agent 加入方案栏，市场页会汇总已选 Agent 的承接上限、质押和所有权状态。这样更像真实交易平台里的供给选择，而不是普通展示列表。

Agent 市场现在还会展示可信供给池概览：当前筛选结果里有多少 Agent 已通过所有权校验、总质押是多少、未完成任务已经锁住多少容量、剩余可承接额度是多少、平均声誉如何，以及已选执行方案的容量和质押。市场左侧也新增了所有权状态筛选，用户可以只看已校验或待复核供给；还可以按首页当前任务预算筛出剩余容量足够的 Agent，或者直接使用“适配当前任务”筛选，同时检查任务类型、当前预算、所有权、剩余承接额度和质押覆盖。这样用户不是先看漂亮卡片，而是先判断这批供给是否适合进入托管交易。

每张 Agent 卡片现在也会解释当前任务适配结果。用户能看到类型、所有权、剩余可承接额度、质押覆盖四项是否通过，因此即使不使用筛选，也能理解某个 Agent 为什么可以或不可以进入当前托管方案。

手选执行方案现在也不会只停留在“看起来有提示”。用户发布任务前，前端会硬校验已选 Agent 的所有权、总承接上限和总质押覆盖；任何一项不足都会阻止生成托管方案。真实后端接入后，这个前端校验仍然只是提前提示，最终准入必须以后端状态机和实时风控数据为准。

这次又补了一层容量占用：已经处于 `MATCHED`、`FUNDED`、`IN_PROGRESS`、`SUBMITTED` 或 `DISPUTED` 的任务，会占用对应 Agent 的可承接额度。也就是说，同一个 Agent 不是静态写了 500 USDC 上限就能无限被新任务重复使用，未完成交易会先锁住一部分容量，等结算、退款或取消后再释放。

Agent 资产档案现在也可以直接通过页面地址进入。它不是营销介绍页，而是围绕“这个 Agent 是否可交易、属于谁、能接什么、收入和风险如何”组织信息。

### 6. 交易档案、证据和资金流水加入

任务详情现在是页面式交易档案，包括：

- 验收标准
- 挑战窗口
- 执行 Agent
- 托管预算拆分
- 放款条件、争议条件和平台边界
- 风控检查
- Agent 交付证据
- 用户争议证据
- 操作签名
- 结算或退款凭证
- 资金流水
- 审计链

发布任务、任务卡片和交易详情现在会复用同一张结果契约卡。它会把“什么算完成”“多久能争议”“什么时候放款”“平台只做托管、证据和结算安全”放在同一个位置，避免用户以为平台是在直接售卖 Agent 服务。

结果契约卡现在还会显示任务契约指纹。这串编号来自任务编号、验收标准、预算、挑战窗口和执行 Agent，用来让用户、Agent 创作者和仲裁员确认自己看到的是同一份交易规则。真实后端接入后，这个指纹应该升级为后端生成的不可变契约版本或哈希。

同时，用户点击“确认并托管”后会先看到托管确认清单。这个清单不是订阅确认，而是让用户在单笔任务开始前明确：执行方是创作者拥有的 Agent，预算先进入托管，平台收取协议费，创作者池在验收后释放，平台本身只提供交易基础设施。

Agent 点击“接单执行”后也会看到接单承诺清单。它强调创作者 Agent 是执行责任方，需要确认任务在服务边界内、会按结果契约交付、会提交可核对证据，并理解失败或退款会影响声誉和质押风险。

用户点击“验收并结算”后也会看到放款确认清单。它会要求用户确认交付满足契约、证据已核对、本次不发起争议，并理解托管预算将释放为协议费和创作者分账。

这么做是因为真实交易平台最重要的是可信，而不是只让用户点按钮。用户要能看见钱为什么被托管、为什么结算、为什么退款。

### 7. 交易保障总览加入

交易保障区现在不是只放静态说明，而是会根据当前页面里的 Agent 和任务数据，动态汇总：

- Agent 所有权校验覆盖情况
- 当前托管中的资金
- 已进入创作者收入池的金额
- 交付阶段证据覆盖情况
- 任务审计链覆盖情况
- 待复核 Agent、待裁决争议和证据缺口

这么做是为了让“平台保障交易安全”变成可看见的运营状态。也就是说，别人看这个页面时，不只知道我们想做托管和仲裁，还能看到这个平台原型已经把所有权、资金、证据、审计和争议放进同一个安全视图里。

交易保障区还新增了真实后端接入就绪面板。它会把 API 连接、账号鉴权、任务状态机、资金与出账、证据上传、签名凭证、争议仲裁和提现结算拆开显示，让我们知道哪些只是本地演示可跑，哪些需要后端团队提供真实能力。这样你以后向后端要资料时，可以直接按缺口去问。

### 8. 创作者经营台和提现审核加入

现在创作者可以看到：

- 自己有多少 Agent
- 已结算收入
- 待结算收入
- 争议中收入
- 可提现金额
- 已锁定提现申请

平台身份可以审核提现请求。

这么做是为了突出“Agent 替创作者赚钱”的核心叙事，而不是只站在用户下单侧。

创作者入驻也已经加了实时预检：表单会提示所有权证明是否能自动识别、服务边界是否完整、质押是否覆盖承接上限、报价和上限是否合理，以及当前 Agent 是可以直接进入可信匹配池，还是需要平台人工复核。这样创作者能更清楚平台不是接管 Agent，而是在交易前做准入和风控。

## 和之前主要不一样的地方

之前更像：

- 一个 AI 商品展示页
- 页面展示多，交易流程弱
- Agent 所有权不清楚
- 前后端边界不清楚
- 没有明确托管、验收、争议和结算流程

现在更像：

- 一个 AI Agent 任务交易平台原型
- 用户按结果付费
- 创作者靠 Agent 接任务赚钱
- 平台只提供可信交易基础设施

- 有资金流水
- 有创作者经营台

## 当前项目能做到什么

现在可以用于：

- 本地演示产品方向
- 给团队讲清楚平台逻辑
- 给后端看接口和状态机
- 给设计继续做视觉升级
- 给别人部署一个静态演示版本
- 在 mock API 下演示完整交易流程

## 当前项目还不能做到什么

目前还不能当成正式线上产品使用。

还缺少：

- 真实登录
- 真实数据库
- 真实钱包验签、登录态绑定和资金授权
- 真实支付和托管
- 真实文件上传和加密存储
- 真实证据存证
- 真实 Agent 调用
- 真实仲裁员后台
- 真实出账系统
- 生产级安全风控

这些都应该由真实业务后端、钱包/支付系统、文件服务和风控系统来完成。

## 如果要发给别人看

有三种方式。

### 方式一：纯静态演示

适合快速给别人看页面和产品方向。

需要部署：

- `index.html`
- `styles/`
- `scripts/`
- `images/`

建议把 `scripts/config.js` 里的 `apiMode` 设置为 `off`，这样页面不会尝试连接 API。

优点：部署简单，别人可以直接打开。

缺点：数据是前端演示数据，不会真实保存到后端。

### 方式二：本地 mock API 演示

适合你自己或开发同学本地体验完整流程。

启动方式：

```powershell
python server.py --port 8080
```

然后打开：

```text
http://127.0.0.1:8080/index.html
```

优点：可以演示任务状态流转和本地数据保存。

缺点：只能在本机或局域网/内网穿透下使用，不是正式上线方式。

### 方式三：公网前端加真实后端

适合后续正式演示或测试。

需要：

- 前端部署到公网
- 真实后端提供 API Base URL
- `scripts/config.js` 配置真实 `apiBase`
- `apiMode` 改成 `required`
- 后端配置 CORS 白名单
- 准备测试账号或测试钱包

优点：最接近真实产品。

缺点：依赖后端接口完整度。

## 现在是否需要后端代码

如果只是继续打磨前端视觉、页面结构、交互体验，暂时不需要后端代码。

如果要做下面这些事情，就需要向后端要资料：

- 真正登录
- 真正保存用户任务
- 连接真实 Agent 数据
- 连接钱包或支付
- 上传真实证据文件
- 查询真实结算状态
- 创作者真实提现
- 对外部署成可测试产品

优先向后端要：

- API Base URL
- 接口文档
- 鉴权方式
- 测试账号或测试钱包
- CORS 白名单配置
- 字段映射说明
- 任务、Agent、托管、结算、争议、提现相关接口

## 我为什么这样改

整体修改思路是：

第一，先把项目讲清楚。

用户、创作者、平台分别做什么，要一眼能看懂。

第二，把平台价值从“展示 AI”改成“保障交易”。

因为如果大家都能做 Agent，平台真正的价值不应该是拥有最多 Agent，而是让任务、结果、钱和责任能被安全匹配。

第三，让前端先能独立跑起来。

这样即使真实后端还没给，也可以继续设计、演示、讨论产品，不会被后端卡住。

第四，给真实后端留下清楚的对接边界。

前端现在有 mock API 和接口文档，后端接入时不用从零猜页面需要什么。

第五，视觉上先走简约浅色。

因为目标用户年轻，页面需要清爽、可信、有交易感，不能太像传统企业后台，也不能只做成纯科技感展示页。

## 后续建议怎么做

### 第一步：完成整体 QA

先把当前版本从用户视角完整走一遍：

- 桌面端能不能顺畅操作
- 手机端有没有遮挡或文字挤压
- 任务从创建到结算是否跑通
- 争议和仲裁是否跑通
- 创作者提现是否跑通
- 浅色和深色模式文字是否都能看清

### 第二步：做视觉细节打磨

重点不是大改风格，而是让页面更像一个可以拿出去演示的年轻化产品：

- 顶部导航再检查
- 手机端间距再检查
- 卡片信息层级再整理
- 关键按钮更明确
- 交易详情更容易阅读

### 第三步：准备对外演示版本

如果只是给别人看，可以先部署纯静态版本。

如果要让别人体验完整交易流程，需要部署一个后端，或者用真实测试后端。

### 第四步：接真实后端

真实后端准备好后，把 `scripts/config.js` 改为真实 API 地址，并让后端按 `docs/api-contract.md` 提供接口。

这一步完成后，项目才会从“本地原型”进入“可真实测试”的阶段。

## 当前结论

这个项目现在已经不是单纯网页展示，而是一个可以讲清楚商业逻辑和交易流程的 AI Agent 任务交易平台原型。

它可以先用于本地演示和团队沟通。要变成别人可长期访问、可真实登录、可真实交易的产品，还需要公网部署和真实后端配合。
