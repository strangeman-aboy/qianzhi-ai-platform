# 对外演示和部署说明

这份文档回答一个核心问题：本地能打开的网页，怎样让别人也能打开。

当前项目可以按三种方式对外展示，复杂度从低到高。

## 方案一：纯静态演示

适合场景：

- 只给别人看页面、产品方向和交互原型。
- 不要求真实登录、真实任务数据、真实托管或结算。
- 希望最快拿到一个公网链接。

需要部署的文件：

- `index.html`
- `styles/`
- `scripts/`
- `images/`

可以放到任意静态站点服务，比如公司服务器、对象存储/CDN、Vercel、Netlify、GitHub Pages 或 nginx 静态目录。

推荐配置：

```js
window.QIANZHI_CONFIG = {
    apiBase: '',
    apiMode: 'off',
    environmentName: '静态演示'
};
```

说明：

- `apiMode: 'off'` 表示页面不请求后端，所有任务状态只在浏览器里本地演示。
- 刷新页面后，本地临时操作不会作为真实业务数据保存。
- 这种方式适合发给投资人、合作方或团队看方向，但不能当真实交易平台使用。

## 方案二：静态网页 + 本地 mock API 演示

适合场景：

- 内部演示完整交易流程。
- 需要展示任务状态流转、签名审计、争议、结算和创作者台账。
- 暂时没有真实后端，但希望数据能在演示机上持久保存。

启动方式：

```powershell
python server.py --port 8080
```

访问：

```text
http://127.0.0.1:8080/index.html
```

说明：

- 这个地址只在当前电脑上可用。
- 如果要临时给外部人员看，可以用内网穿透工具把 `8080` 暴露成临时公网地址。
- 电脑关机、服务停止或网络变化后，外部链接就可能失效。
- `server.py` 是 mock 服务，不建议作为真实生产后端。

## 方案三：公网静态网页 + 真实业务后端

适合场景：

- 希望别人稳定访问公网域名。
- 需要真实账号、钱包、任务、Agent、托管、争议和结算数据。
- 后端团队已经提供正式或测试 API。

部署结构：

```text
用户浏览器
  -> 前端静态站点：https://www.example.com/ai-platform/index.html
  -> 真实后端 API：https://api.example.com/api/...
```

前端配置：

```js
window.QIANZHI_CONFIG = {
    apiBase: 'https://api.example.com',
    apiMode: 'required',
    environmentName: '真实业务后端'
};
```

后端需要确认：

- `GET /api/health` 可被公网访问。
- CORS 允许前端域名访问 API。
- 任务状态流转接口兼容 `docs/api-contract.md`。
- 关键交易操作支持登录态、钱包签名、DID 签名或服务端会话签名。
- 交付物、证据、托管、结算和争议数据由后端持久化。

## 上线前检查清单

- 页面公网地址可以打开。
- `scripts/config.js` 指向正确环境。
- 静态资源路径正常，logo、CSS、JS 都能加载。
- 如果是静态演示，`apiMode` 设置为 `off`。
- 如果是真实后端，`apiMode` 设置为 `required`，避免悄悄退回本地演示数据。
- 真实后端已配置 CORS 白名单。
- 钱包、登录、签名、托管、争议、结算接口都有测试账号或测试钱包。
- 页面顶部运行环境状态栏能让演示者判断当前连的是静态演示、本地 mock API、测试后端还是正式后端。
- 正式演示真实交易前，状态栏必须显示真实后端连接成功，不能是 `静态演示`、`本地 Mock API` 或 `API 必连失败`。

## 给别人看的推荐路径

当前阶段建议先用“纯静态演示”或“静态网页 + 测试后端”。

等真实后端接口稳定后，再切换到“公网静态网页 + 真实业务后端”。这样前端可以先承担产品表达和交易流程验证，后端再逐步接管真实身份、真实资金和真实数据。
