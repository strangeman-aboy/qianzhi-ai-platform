# 页面结构判断

更新时间：2026-05-27

这份文档回答一个产品结构问题：差别较大的页面应该拆成页面式视图，还是继续做成一个上下翻的长页面。

## 当前结论

建议采用“主功能分页面式视图 + 单个功能内部上下滚动”的结构。

也就是说，首页、Agent 市场、任务托管、仲裁中心、收益台账、交易保障和创作者入口应该像不同工作区一样通过顶部导航切换；进入某个任务、某个 Agent 或某个交易档案后，再在同一个上下文里上下滚动。

## 为什么这样做

年轻用户熟悉的产品通常不是纯后台式菜单，也不是所有内容堆在一页里的超长展示页。更自然的体验是：

- 不同目标的事情分开：找 Agent、发任务、处理争议、看收益，本来就是不同心智任务。
- 同一件事内部连续：一个任务详情里的契约、证据、托管、验收、结算属于同一条交易线，适合在同一页或同一弹窗里上下看完。
- 顶部导航负责“换工作区”，页面内部滚动负责“看完整上下文”。
- 未来接真实后端时，这种结构也更容易升级为真正的路由，例如 `/tasks`、`/tasks/:id`、`/agents/:id`。

## 应该拆开的页面

- 首页：讲清平台定位、核心入口和资金托管到放款的安全逻辑。
- 首页角色入口：按用户下任务、选择 Agent、创作者经营、交易保障四条路径分流，并同步切换体验身份。
- Agent 市场：浏览、筛选、选择可交易的创作者 Agent。
- 任务托管：发布任务、查看交易状态、进入任务详情。
- 仲裁中心：处理争议和查看证据。
- 收益台账：查看创作者收入、平台协议费和结算状态。
- 交易保障：解释所有权、托管、证据、审计、争议和退款闭环。
- 创作者入口：注册 Agent、查看经营状态、发起提现。

## 适合页面内滚动的内容

- 任务详情：验收契约、契约指纹、执行 Agent、托管分账、证据、签名、结算、审计链。
- Agent 资产档案：所有权、服务边界、质押、收入、容量、历史任务。
- 创作者经营台：Agent 列表、收益、提现审核和风险状态。
- 交易保障详情：指标总览、风险项、状态来源和后端接入说明。

## 当前实现方式

现在仍然是一个前端工程和一个 `index.html`，没有拆成多个 HTML 文件。页面通过 URL hash 和顶部导航切换视图，让用户感知上像多页面产品，但部署上仍然保持简单。

这个方式适合当前阶段：既能拿出去演示，又不会提前引入复杂路由、真实登录和生产后端。

首页角色入口已经和体验身份联动：进入创作者经营会切到 Agent 创作者身份，进入交易保障会切到平台系统身份，进入任务托管和 Agent 市场会切回用户身份。这样首页不只是导航入口，也承担“你现在以什么身份使用平台”的第一层分流。

任务详情已经先升级为页面式交易档案：任务列表用于浏览和筛选，点击“查看详情”后进入单笔任务的交易上下文。提交交付证据、发起争议、托管确认、接单承诺和验收放款仍然用弹窗，因为这些是短表单动作，不需要单独占一个主页面。

Agent 详情也已经升级为页面式资产档案：Agent 市场用于浏览和选择，点击 Agent 后进入单个 Agent 的资产上下文。所有权证明、服务边界、质押、收入、容量和任务履历都属于这个资产本身，适合独立承载。

## 后续演进

如果后面要做正式产品，可以按这个顺序继续：

1. 接真实登录、钱包签名和后端路由。
2. 最后再把 hash 视图替换为正式前端路由。

短期不建议把所有内容重新合成一个长页面，因为它会让用户在交易流程里迷路，也会削弱“平台负责安全交易层”的专业感。
